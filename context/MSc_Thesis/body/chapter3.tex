\section{NegotiationArena Framework} \label{sec:na-framework}

\textcite{bianchi_how_2024} introduced NegotiationArena, an open-source framework to evaluate and analyze the negotiation abilities of LLM-based agents. The original work focused on proprietary models, some of which have since been deprecated; we keep the games and protocol unchanged, but replace the agents with the open-weight models identified in Table~\ref{tab:experiment-models} and extend it by implementing two inference-time techniques (Sections~\ref{sec:self-refine} and~\ref{sec:team-negotiation}). 

\subsection{Games} \label{sec:games}

In each scenario, two agents alternate turns for up to a fixed number of rounds, or until one side terminates the game. Before each move, the agent reasons privately step by step; this reasoning is never revealed to the opponent.

\paragraph{Resource Exchange Scenario (Trading game).}
In the trading game, each party starts with a set of resources and aims to maximise its total holdings. The parties exchange proposals until one is accepted or the maximum number of rounds is reached. The agent with the greater gain wins.

\paragraph{Multi-Turn Ultimatum Game.}
In this variation of the classic Ultimatum Game \cite{doi:10.1126/science.1082976}, one agent receives all of the game's resources but must propose a split that the other accepts in order for either to keep any of it. The second agent can either accept, counter-propose a new split, or reject outright, causing both agents to lose all resources. If no agreement is reached within the maximum number of rounds, both agents lose everything. The winner is the agent with the higher share of the agreed split.

\paragraph{Seller and Buyer Scenario.}
This is an incomplete-information game in which a seller and a buyer negotiate the price of an item. Each party holds a private valuation: the seller knows its production cost and the buyer its willingness to pay. On each turn, a player may issue a \textsc{Proposal}, \textsc{Accept} the opponent's offer, or \textsc{Reject} it (ending the game with no deal). An agent wins if the agreed price is closer to the opponent's valuation than to its own.

\subsection{Evaluation Metrics}

Negotiation performance is measured using the two metrics adopted in the NegotiationArena (Win Rate and Average Payoff), complemented by a Completion Rate.

\begin{itemize}
    \item \textbf{Win Rate:} Percentage of \textit{completed} games in which a party obtained a higher payoff than the other. As in NegotiationArena, ties are ignored.
    \item \textbf{Average Payoff:} Average number of resources of each agent after the trade. Its definition is game-specific: in Price Negotiation, it is the surplus ($\text{price} - \text{cost}$ for the seller, $\text{value} - \text{price}$ for the buyer); in Trading, it is the net change in resource value ($\text{final} - \text{initial}$); and, in Ultimatum, it is the number of dollars retained.
    \item \textbf{Completion Rate:} Proportion of games that reach a final state. Because smaller open-weight models frequently violated the response format, this metric is reported alongside the others and is the primary metric for the retry ablation.
\end{itemize}

\section{Models and Infrastructure} \label{sec:models-infra}

Open-weight models vary considerably in scale, so our objective was to group them into comparable parameter tiers in order to measure performance differences across model sizes. We defined three tiers: 4–9B, 12–14B, and 24–27B parameters. 

For each tier, we selected models from three families: Qwen (with thinking mode disabled), Mistral, and Gemma. Maintaining a consistent version across all tiers was not possible for Qwen and Mistral, so we included models from both Qwen 3 and Qwen 3.5, and from Ministral and Mistral Small, respectively.  We initially considered Llama models as well, but observed very low adherence to the required output format with this family of models. Given the additional computational overhead this would entail, we excluded them from the benchmark. The selected models are listed in Table~\ref{tab:experiment-models}.

We applied $8$-bit quantization (\texttt{bitsandbytes}\footnote{\url{https://github.com/bitsandbytes-foundation/bitsandbytes}}) to Gemma, Qwen, and Mistral-Small models; the Ministral models are already distributed in native FP8 precision. Generation uses temperature $0.7$ and top-p $0.9$ sampling, with a maximum of $1024$ new tokens per generation call.

We implemented a base agent built around the HuggingFace \texttt{transformers} library\footnote{\url{https://huggingface.co/docs/transformers/en/index}}, abstracting model loading, quantization, and chat-template formatting behind a single interface. This design kept the experiments agnostic to the underlying model family and execution environment, allowing the same code to run unchanged across all three platforms.

Runs were distributed across three environments: Kaggle (2$\times$ NVIDIA T4), the MIA server (NVIDIA L40S), and the Deucalion HPC cluster (NVIDIA A100) \footnote{\url{https://rnca.fccn.pt/en/deucalion/}}.

\section{Benchmarking Open-Weight LLMs} \label{sec:benchmarking}

The first contribution follows the experimental framework of NegotiationArena \cite{bianchi_how_2024}, evaluating open-weight models in place of the proprietary models of the original study. Our goal is to establish a baseline of how well current open-weight models negotiate across the three games and parameter tiers. Additionally, following \textcite{bianchi_how_2024}, we examine whether assigning a social persona affects negotiation outcomes.

\subsection{Model Cross-Play} \label{sec:cross-play}

To measure how well current open-weight models negotiate among themselves, we run a round-robin within each parameter tier. Each ordered pairing is played for 30 games in each of the three scenarios, using the default parameters of Table~\ref{tab:game-setup}. Because the games are role-asymmetric, both orderings of each pair are executed. 

\subsection{Social Personas} \label{sec:personas}

Following \textcite{bianchi_how_2024}, we analyse whether assigning a social persona to one party changes the outcomes of the negotiations. Player 1 always uses the default behaviour, while Player 2 is assigned one of three personas: \textit{baseline} (no persona), \textit{desperate} (instructed to feign desperation and beg for a better deal), or \textit{cunning} (instructed to be manipulative and to insult the opponent). The persona text is appended directly to Player~2's game system prompt; the exact wording for each persona and game is listed in Table~\ref{tab:persona-prompts}. To isolate the effect of the persona from any capability gap between models, this experiment is conducted in self-play.

\subsection{Self-Correction} \label{sec:retry-ablation} 

A practical obstacle with smaller open-weight models was that they frequently produced responses violating the required tag format, which could not be parsed into a valid move, thereby ending the game prematurely. To address this, we assess whether a simple error detection followed by a retry mechanism could improve the completion rate. When a response fails to parse, the agent receives an error message containing the invalid response and the corresponding parser error, and is asked to resubmit its move in the correct format; this repeats up to \texttt{max\_retries} times, after which the game ends unparsed and is counted as incomplete. Figure~\ref{fig:retry_loop} illustrates this loop. We compare the baseline (\texttt{max\_retries}\,$=0$) against \texttt{max\_retries}\,$=3$ across both experiments.

A move is committed only once it parses into the expected tag format; otherwise, the agent is fed the parser error and asked to re-generate the response, up to \texttt{max\_retries} times, before the game is considered as incomplete. \section{Self-Refine} \label{sec:self-refine}

% Citar prompts que estão no anexo;

We then investigate whether a model negotiates better when given the chance to critique and rewrite its own move before committing to it. As introduced in Section~\ref{sec:self-refinement}, we apply the Self-Refine technique of \textcite{madaan_self-refine_2023}: rather than immediately committing to an answer, the agent cycles through a \textit{draft $\rightarrow$ feedback $\rightarrow$ refine} loop. Figure~\ref{fig:self_refine_loop} illustrates the adapted loop.

\begin{figure}
    \centering
    \includegraphics[width=0.8\linewidth]{figures//methodology/SelfRefine___Figure.jpg}
    \caption{Self-Refine technique. After generating an initial draft (Phase 1), the agent critiques its output along five criteria (Phase 2). Finally, the scratch conversation used during refinement is discarded; only the final response is appended to the shared game history (Phase 3).}
    \label{fig:self_refine_loop}
\end{figure}

Whenever an agent is asked to choose a move, it first produces an initial draft and then runs two critique-and-rewrite iterations. Each iteration prompts the agent to critique its current draft along the following criteria:

\begin{itemize}
\item \textbf{Format:} whether every required tag is present and parseable;
\item \textbf{Payoff Alignment:} whether the proposed trade advances the agent's own valuation or goal;
\item \textbf{Opponent Plausibility:} whether the move is likely to be accepted given the dialogue so far;
\item \textbf{Consistency:} whether the reasoning, message, and move agree with one another;
\item \textbf{Rule Compliance:} whether the proposal budget and turn limits are respected.
\end{itemize}

A second \textit{refine} prompt then asks the model to rewrite the response, incorporating the critiques. The critique and intermediate drafts are produced on a scratch copy of the conversation; once refinement ends, this scratch copy is discarded, and only the final response is appended to the shared game history.

We evaluate the four combinations obtained by assigning each party either the default or the Self-Refine strategy: both default, both Self-Refine, and the two mixed conditions. As in the social-persona experiment (Section~\ref{sec:personas}), every condition is run in self-play; each condition is played for 30 games across all three scenarios and model tiers. The feedback and refinement prompts are given in Appendix~\ref{ap1:sr-prompts}.

Unlike the previous experiments, each move requires five model calls: one initial draft plus, for each of the two iterations, one feedback and one rewrite call (Table~\ref{tab:inference-cost}). Whether this increase in inference cost yields a measurable improvement in negotiation performance is one of the main questions addressed in the results chapter.

\section{Team Negotiation} \label{sec:team-negotiation}

The final contribution replaces a single negotiation party with a \textit{team} of \textit{N} models that debate internally before emitting one move. Earlier (pre-\acs{LLM}) research in automated multi-agent systems has shown that team-based negotiation strategies generally outperform those of single agents \cite{S_nchez_Anguix_2013}. In parallel, recent multi-agent work shows that letting several language models propose, critique, and vote on a shared answer can raise quality above that of any single member, whether through round-table consensus \cite{chen2024reconcileroundtableconferenceimproves} or debate \cite{du2023improvingfactualityreasoninglanguage}. We bring these ideas together and ask whether a deliberating team of LLMs negotiates better than a single agent. 

\begin{figure}
    \centering
    \includegraphics[width=0.8\linewidth]{figures/methodology/team_based_negotiation_fig.png}
    \caption{Team deliberation process. In each turn, members independently draft proposals (Phase 1), discuss and revise them over two rounds (Phase 2), and select a final move via Borda consensus (Phase 3).}
    \label{fig:team_negotiation}
\end{figure}

We represent one party by a team of \textit{N} members and let them deliberate privately before emitting a move. Each turn is divided into three phases, \textit{draft $\rightarrow$ discuss $\rightarrow$ consensus}, as illustrated in Figure~\ref{fig:team_negotiation}. 

In the \textbf{drafting} phase, each member independently proposes a possible move, using the same prompt a single agent receives (following the same design as previously described for the single-agent architectures). Consequently, after the first phase, the team ends with up to \textit{N} distinct proposals.

In the \textbf{discussion} phase, every member is shown their own draft alongside their teammates' proposals. Each member then keeps, merges, or adopts whichever proposal they judge strongest for the team and re-emits a full move. The discussion prompt is given in Appendix~\ref{ap1:team-prompts}.

Finally, in the \textbf{consensus} phase, the revised drafts form a slate of candidate moves (one draft per member) and each member ranks this entire set from best to worst (Listing~\ref{lst:team-ranking}). A Borda count over these rankings selects the move: in a ranking of $s$ candidates, the candidate in the top position ($k=0$) earns $s-1-k = s-1$ points, the next $s-2$, and so on down to $0$ for the last. The move with the highest total wins. In case of a tie, the move with the lowest index wins. Unlike the Similarity-Based Borda Voting strategy \cite{S_nchez_Anguix_2013}, accepting the opponent's offer is not handled separately; it is simply one of the moves a member can draft, so a single vote decides both \textit{what} to propose and whether to \textit{accept}, with no separate voting phase to determine acceptance.

The entire deliberation process (i.e., drafting, discussion, and ranking prompts) is run privately. Once the team's move is chosen, the history is rolled back to the initial snapshot, and only the agreed move is appended to each member's history. The opposition party, therefore, sees exactly one move per turn, with none of the internal deliberation process. One complication follows from this rollback: the reasoning attached to the winning move was written in the context of the discussion and may refer to teammates that are no longer visible once the deliberation is erased. To keep each member's private history coherent on subsequent turns, before committing the move, the winning member rewrites its \texttt{<reason>} block into a first-person reasoning that mentions no teammates, candidates, ranking, or ``team'', while keeping the same strategy and conclusion (Listing~\ref{lst:team-reason}).

To understand the impact of diversity in team-based negotiations, we compare \textbf{homogeneous} teams, made of \textit{N} copies of a single model, against \textbf{heterogeneous} teams whose members are drawn from different model families. We fix the team size at $N = 3$ and the number of discussion rounds at $R = 2$. For this experiment, we only analysed the models in the small category (12--14B). The heterogeneous team is composed of one model from each family.  

A homogeneous team plays only against a single agent of its own model: a team of three Gemma-based agents faces a single Gemma-based agent, a team of three Qwen-based agents faces a single Qwen-based agent, and a team of three Mistral-based agents faces a single Mistral-based agent. The heterogeneous team matches against each of the three families in turn.

As with Self-Refine, this deliberation comes with extra costs. A turn costs $N$ drafting calls, $N \times R$ revision calls during discussion, $N$ ranking calls, and one reasoning-rewrite call, for a total of $N(2+R)+1$ calls. In our experiments, $N = 3$ and $R = 2$, so there is a thirteen-fold increase in the number of calls compared with the default (Table~\ref{tab:inference-cost}).
